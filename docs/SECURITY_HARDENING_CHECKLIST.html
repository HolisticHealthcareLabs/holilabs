Executive Summary

On March 2026, an autonomous AI agent (Claude, Anthropic) discovered and exploited a critical remote code execution vulnerability in the FreeBSD kernel (CVE-2026-4747) in approximately 4 hours of compute time, at a cost of a few hundred dollars. This represents a paradigm shift: offensive cyber capabilities previously exclusive to nation-state actors are now commoditized. The researcher's pipeline has already identified 500+ high-severity vulnerabilities across various codebases.

HoliLabs v2 handles Protected Health Information (PHI) under three regulatory frameworks (HIPAA, LGPD, ANVISA). Our current security posture has strong foundations — Helmet.js CSP, Pino log redaction for PII fields (CPF, SSN, email), structured auth middleware, and CI security scanning — but critical gaps remain. Rate limiting is disabled (lines 96-102, apps/api/src/index.ts). JWT token lifetimes are not optimized for the new threat window. Column-level encryption for PHI is absent. The CI security workflow runs on manual trigger only.

This checklist prioritizes 17 hardening items across three tiers: ship-blockers (must complete before GTM), week-1 post-launch, and month-1 strategic. Total estimated effort is 231 engineering hours. The threat model assumes AI-speed attacks against our API surface, supply chain, and authentication layer.

Threat Model: AI-Accelerated Attacks

The CVE-2026-4747 incident proves that the attacker-defender time asymmetry has inverted. Key parameters for HoliLabs:

Attack surface: 11 Fastify route modules, 49KB+ Prisma schema, 100+ npm dependencies
AI exploit time: ~4 hours | Our current patch time: Unknown (no automated rollback)
PHI exposure risk: Patient records, CPF, medical diagnoses, lab results, prescriptions
Regulatory penalty: LGPD: up to 2% of revenue | HIPAA: up to $1.9M per violation category

CRITICAL Priority 1 — Ship Blockers (Before GTM)

These 6 items must be completed before any production deployment. Each addresses a direct exploit vector that AI-speed attacks would target within hours of discovery.

1.1 Enable API Rate Limiting Critical
Current state: Rate limiting is imported but DISABLED (lines 96-102 in apps/api/src/index.ts). Comment reads: "Rate limiting temporarily disabled for local development. TODO: Implement proper Redis client with ioredis."
// CURRENT (disabled): // await server.register(rateLimit, { max: 100, timeWindow: '15 minutes', redis: env.REDIS_URL }); // TARGET: await server.register(rateLimit, { max: 100, // General: 100 req/min timeWindow: '1 minute', redis: new Redis(env.UPSTASH_REDIS_REST_URL), // Upstash (already in deps) allowList: ['127.0.0.1'], // Dev bypass keyGenerator: (req) => req.headers['x-forwarded-for'] || req.ip, }); // Auth endpoints: 10 req/min | Password reset: 5 req/min // Add route-level overrides via { config: { rateLimit: { max: 10 } } }
Effort: 4h | Owner: Backend | Dependencies: UPSTASH_REDIS_REST_URL env var
1.2 JWT Token Hardening Critical
Why: The FreeBSD exploit window was 4 hours. If our JWT access tokens live longer than 15 minutes, a stolen token provides an attacker more time than needed to exfiltrate all PHI.
// Access token: 15 min expiry (down from default) // Refresh token: 7 days, single-use rotation // Device binding: hash(User-Agent + IP-subnet) stored in token claims // On refresh: validate fingerprint matches; if not, force re-auth // Implementation: jose library (already in deps)
Effort: 8h | Owner: Backend | Files: auth middleware, token service, refresh endpoint
1.3 Input Validation on All API Routes Critical
Why: The FreeBSD exploit crafted multi-packet shellcode through unchecked input. Zod is already in our deps but not enforced on every route. An AI fuzzer can probe all 11 route modules in minutes.
// Audit checklist for each route module: // apps/api/src/routes/auth.ts — ✅ Has Zod? Verify body + params // apps/api/src/routes/patients.ts — ❓ Check all CRUD operations // apps/api/src/routes/ai.ts — ❓ AI endpoints accept freeform text — validate length // apps/api/src/routes/upload.ts — ❓ File type + size validation // apps/api/src/routes/exports.ts — ❓ Query params validation // apps/api/src/routes/admin.ts — ❓ Admin operations need strict schemas // apps/api/src/routes/telemetry.ts — ❓ Sidecar data ingestion // apps/api/src/routes/monitoring.ts — ✅ Read-only // apps/api/src/routes/fhir-admin.ts — ❓ FHIR payloads are complex
Effort: 16h | Owner: Backend | Pattern: Zod schema per route, fastify-type-provider-zod
1.4 CSP & Security Headers Tightening Critical
Current state: Helmet is configured (lines 79-88) but CSP allows 'unsafe-inline' for styles. Missing: Permissions-Policy, HSTS preload, Referrer-Policy.
// Tighten Helmet config: await server.register(helmet, { contentSecurityPolicy: { directives: { defaultSrc: ["'self'"], styleSrc: ["'self'", "'unsafe-inline'"], // TODO: Remove unsafe-inline with nonce scriptSrc: ["'self'"], // GOOD: No unsafe-eval imgSrc: ["'self'", 'data:', 'https://storage.googleapis.com'], // Pin CDNs connectSrc: ["'self'", env.API_URL, env.MEDPLUM_URL], frameSrc: ["'none'"], objectSrc: ["'none'"], }, }, hsts: { maxAge: 63072000, includeSubDomains: true, preload: true }, referrerPolicy: { policy: 'strict-origin-when-cross-origin' }, permissionsPolicy: { features: { camera: ["'none'"], microphone: ["'self'"], geolocation: ["'none'"] } }, });
Effort: 4h | Owner: Backend/DevOps | Risk: May break LiveKit if camera/mic blocked — test telehealth flow
1.5 CORS Origin Lockdown Critical
Current state: CORS origin is set from env.CORS_ORIGIN (line 75). Verify this is NOT set to * in production. Must be explicit domain list.
// Ensure .env.production has: // CORS_ORIGIN=https://app.holilabs.xyz,https://dashboard.holilabs.xyz // NOT: CORS_ORIGIN=* (allows any origin to make authenticated requests)
Effort: 2h | Owner: DevOps | Verify: All environments (.env.staging, .env.production)
1.6 CI Security Scanning on Every PR Critical
Current state: .github/workflows/security-enhanced.yml runs on workflow_dispatch (manual trigger only). This means security scans only happen when someone remembers to run them. In the AI-speed era, every PR must be scanned.
# Change trigger from: on: workflow_dispatch # To: on: pull_request: branches: [main, develop] push: branches: [main] schedule: - cron: '0 6 * * 1' # Weekly Monday scan
Effort: 2h | Owner: DevOps | Add: npm audit --audit-level=high as blocking step

HIGH Priority 2 — Week 1 Post-Launch

2.1 Supply Chain Scanning (Socket.dev / Snyk) High
Why: 100+ npm deps including medical libraries (Medplum, Cornerstone.js, Deepgram). The xz-utils supply chain attack pattern could target any of these. Socket.dev detects behavioral anomalies in packages (not just known CVEs).
Effort: 8h | Owner: DevOps | Action: Add Socket.dev GitHub App + CI integration
2.2 Column-Level Encryption for PHI High
Why: Prisma schema has CPF, phone, email, patient diagnoses, lab results stored as plaintext strings. Supabase provides at-rest encryption, but a SQL injection or leaked connection string exposes everything. Application-level encryption means even a full DB dump is useless.
// Fields requiring app-level encryption (from schema.prisma): // Patient: cpf, phone, emergencyContactPhone, emergencyContactName // User: phone, cpfEncrypted (exists but verify implementation) // LabResult: value, referenceRange (clinical data) // Diagnosis: icdCode, description (clinical data) // Prescription: all fields (medication, dosage, instructions) // // Implementation: prisma-field-encryption or custom AES-256-GCM middleware // Key management: AWS KMS or environment variable (ENCRYPTION_KEY)
Effort: 24h | Owner: Backend | Migration: Encrypt existing data in-place, add decrypt-on-read middleware
2.3 Query Rate Limiting at ORM Level High
Why: API rate limiting protects the HTTP layer. But a compromised session could issue expensive queries (SELECT * on patient tables) through legitimate endpoints. Add Prisma middleware that limits query volume per session.
Effort: 8h | Owner: Backend | Pattern: Prisma $use() middleware with per-session query counter
2.4 Immutable Audit Log Separation High
Current: AuditLog model exists in Prisma schema. But if an attacker gains DB write access, they can DELETE audit entries to cover tracks. Separate audit logs into an append-only schema with revoked DELETE/UPDATE permissions.
-- Prisma migration: CREATE SCHEMA IF NOT EXISTS audit; ALTER TABLE "AuditLog" SET SCHEMA audit; REVOKE DELETE, UPDATE ON audit."AuditLog" FROM app_user; GRANT INSERT, SELECT ON audit."AuditLog" TO app_user;
Effort: 8h | Owner: Backend/DBA | HIPAA: Required for breach forensics (§164.312(b))
2.5 mTLS for Edge Server Communication High
Why: The edge server (apps/edge/) syncs hospital LAN data to the cloud API. Hospital networks are high-value targets. mTLS ensures both sides authenticate with certificates, preventing MITM attacks even on compromised hospital networks.
Effort: 16h | Owner: DevOps/Backend | Deps: Certificate generation, rotation schedule, ICP-Brasil integration

STRATEGIC Priority 3 — Month 1

3.1 AI-Powered Codebase Security Review in CI Strategic
The same AI capability that found CVE-2026-4747 can defend us. Add a CI step that sends PR diffs to Claude for security analysis — looking for injection vectors, auth bypasses, PHI leaks, and unsafe patterns specific to our stack.
Effort: 16h | Owner: DevOps/AI | Cost: ~$0.50/PR for Haiku analysis
3.2 Runtime Anomaly Detection Strategic
Add a lightweight service watching Prometheus metrics + API logs for patterns that indicate automated probing: sequential endpoint enumeration, De Bruijn-pattern payloads, unusual header combinations, bulk data access from single sessions.
Effort: 24h | Owner: Backend/DevOps | Integration: Prometheus alertmanager → PagerDuty
3.3 Automated Rollback on Error Spike Strategic
If 5xx error rate exceeds 5% for 2 minutes post-deploy, automatically rollback to previous version without human intervention. Currently deploy.yml has no rollback mechanism.
Effort: 12h | Owner: DevOps | Platform: DigitalOcean App Platform rollback API
3.4 FHIR Sync Kill Switch Strategic
If the Medplum FHIR connection is compromised, we need a one-click way to sever all external data flows while keeping the app functional in offline mode. The edge server architecture already supports this — formalize it as an emergency procedure.
Effort: 8h | Owner: Backend | Trigger: Admin API endpoint + PagerDuty action
3.5 Breach Notification Pipeline Strategic
LGPD requires breach notification within 72 hours. HIPAA within 60 days. Build an automated workflow: detect breach → classify severity → generate notification templates → route to legal → track deadlines.
Effort: 16h | Owner: Backend/Legal | LGPD: Art. 48 | HIPAA: §164.408
3.6 Penetration Testing Schedule Strategic
Contract a third-party pentest firm for quarterly assessments. Focus areas: API auth bypass, PHI exfiltration, FHIR injection, Electron sidecar privilege escalation, hospital edge server compromise.
Effort: 40h (coordination) | Owner: CTO/Security | Budget: $15-30K/quarter
